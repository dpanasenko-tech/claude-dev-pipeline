Инварианты, которые держат схему целой

→1 стек ↔ 1 клон. dev1/dev2 никогда не питают stage/prod. Bind-mount = ветка клона молча определяет код стека.
→Deploy-клон всегда на main. Ветку в нём не переключать — иначе общий stage подхватит чужой WIP.
→Staging: тенанты is_active=false. Токены dummy. Иначе cron бьёт внешний API → бан per-token.
→Пересоздавать, не рестартить. up --force-recreate, не docker restart — иначе живой процесс держит старый код.
→gh MERGED ≠ код на main. Чейнинг PR оседает в промежуточных ветках → git grep origin/main до деплоя.
→Окно VS Code = своя папка. Сессия A→dev1, B→dev2, не рут. Deploy-клон не открывать.
Узкое горлышко — не число окружений, а один main + ручные аппрувы + ограниченные ресурсы staging-хоста (потолок ~2 dev-лейна). Параллельность максимальна на Stages 1–5; всё сходится в один main и один staging (Phase B), затем промоушен на prod.

Управление версиями продукта

Ортогонально топологии выше: как номер версии рождается, запекается в образ и становится видимым в рантайме.

① Авторингtag · changelog · release
🔒нет тега → REFUSE
② СборкаARG→ENV→LABEL
→build-args
③ РантаймGET /version
→per-host
④ Потреблениеcheckout тега → build
Тег — единственный источник версииSemVer

Строка версии берётся только из git-тега, руками не придумывается.

git describe --tags --exact-match
тег == OCI label == /version.version
Метаданные запечены в образOCI

Коммит восстановим по образу без запуска контейнера.

image.revisionimage.versionimage.created
docker inspect · GIT_SHA · BUILD_TIME
env — из APP_ENV хостаper-host

Один и тот же тег отличают по окружению полем env.

{version, git_sha, built_at, env}
«vX / staging» ∥ «vX / prod»
Окружения потребляют версию независимокаталог версий

GitHub Releases — общий список всех версий; что где крутится решает каждый хост при деплое (checkout тега → build). Staging может стоять на vX, prod в тот же момент — на vY. Prod — consume-only: только git fetch --tags → build, теги сам не создаёт, git-cliff туда не ставится.

release рождается один раз · prod только потребляет

Инварианты версионирования

→Нет деплоя без SemVer-тега (AC-6). untagged / dirty / latest / unknown → деплой exit 1.
→Релиз рождается один раз. На машине-авторинге (есть gh write + git-cliff). Где именно — dev/CI vs staging — открытый вопрос черновика.
→Prod — consume-only. Никогда не создаёт теги и не авторит релизы. git-cliff на prod не ставится.
→CHANGELOG живёт в git. Коммитится как код; pull перед правкой → нет расхождений между аккаунтами.
→Тег окружение не диктует. Один каталог версий; выбор «что где» — на хосте. Одно ≠ другому.
→git-cliff — инструмент авторинга, не рантайм. Не в pyproject / uv.lock, в образ не попадает.
