# Human Approval Guide

Как делать review и approve каждой стадии PR.

## Где открыть PR

```bash
# Список открытых PR:
gh pr list

# Открыть PR в браузере:
gh pr view <PR-number> --web

# Посмотреть diff в терминале:
gh pr diff <PR-number>
```

Или открыть вкладку Pull Requests в браузере (`gh repo view --web`).

## Запустить Claude review (рекомендуется перед approve)

```bash
# В Claude Code сессии:
/review <PR-number>
```

Читаешь вывод Claude: если есть блокеры — пишешь комментарий в PR, acc исправляет.
Если "Approved" — переходишь к merge.

## Что смотреть в каждой стадии

### Stage 1 — Brief (`brief.md`)

Вопросы:
- Проблема сформулирована конкретно (не абстрактно)?
- Scope ограничен: есть **non-goals**?
- Acceptance criteria измеримы (AC-1, AC-2…)?
- Нет лишних деталей реализации (это ещё не plan)?

Красные флаги:
- AC типа "работает хорошо" — нет метрик
- Scope включает несколько несвязанных вещей
- Нет явного ответа "что НЕ входит в задачу"

---

### Stage 2 — Plan (`plan.md`)

Вопросы:
- Архитектурные решения обоснованы (почему именно так)?
- Изменения БД описаны: таблицы, поля, миграции?
- Внешние зависимости указаны (API, сервисы)?
- Риски и как их митигировать?
- Rollback strategy есть?

Красные флаги:
- "Сделаем как обычно" без обоснования
- Нет описания что меняется в схеме БД
- Не упомянуты edge cases

---

### Stage 3 — Tests (`test-matrix.md` + `tests/`)

Вопросы:
- Тесты **падают** (не зелёные)? Это норма — реализации ещё нет.
- Каждый AC из brief.md покрыт хотя бы одним тестом?
- Тесты тестируют поведение, а не внутреннюю реализацию?
- Мок только то, что мы не владеем (внешние API)?

Как проверить что тесты падают:
```bash
cd "~/projects/acme-platform"
gh pr checkout <PR-number>
scripts/quality-check.sh  # должен упасть на тестах
```

Красные флаги:
- Тесты зелёные (значит тестируют пустышку или уже есть реализация)
- AC из brief не покрыты
- Тесты тестируют `== None` и ничего больше

---

### Stage 4 — Implementation (код в `src/`)

Вопросы:
- Все тесты зелёные?
- Код не выходит за рамки плана (нет scope creep)?
- Нет новых абстракций "на будущее"?
- Логи на границах системы есть?

Как проверить:
```bash
gh pr checkout <PR-number>
scripts/quality-check.sh  # должен быть зелёным
```

Красные флаги:
- Тесты не зелёные
- Код делает больше чем написано в plan.md
- Добавлены зависимости без обоснования

---

### Stage 5 — Review (`review.md`)

Вопросы:
- Блокеры (если есть) описаны конкретно с указанием файла/строки?
- Если блокеры есть — они разрешены или задача вернулась на доработку?
- Review покрывает: correctness, security, performance, observability?

Красные флаги:
- `review.md` говорит "всё хорошо" без конкретики
- Есть unresolved блокеры
- Security-раздел пустой

---

### Stage 6 — Release (`release.md`)

Вопросы:
- Явный GO или NO-GO?
- CI зелёный на merge commit?
- Rollback plan написан?
- Env vars / секреты указаны для target env?

Красные флаги:
- Нет явного GO/NO-GO
- Rollback = "откатим руками"
- Не упомянуты миграции

---

## Как approve и merge

```bash
# Approve PR:
gh pr review <PR-number> --approve

# Squash merge (стадия → 1 коммит в main):
gh pr merge <PR-number> --squash --delete-branch

# Или в браузере: кнопка "Squash and merge"
```

## Если нужны правки

```bash
# Написать комментарий с конкретным описанием что исправить:
gh pr comment <PR-number> --body "Stage 2: в plan.md не описан rollback для миграции таблицы orders. Добавь раздел Rollback."

# acc исправит и запушит в ту же ветку — PR обновится автоматически.
```

## Быстрый чеклист перед merge

- [ ] Claude `/review` прошёл или я сам прочитал diff
- [ ] Артефакт стадии соответствует ожиданиям (см. таблицу выше)
- [ ] CI зелёный (GitHub Actions badge в PR)
- [ ] Нет unresolved review comments
- [ ] Merge strategy: squash
