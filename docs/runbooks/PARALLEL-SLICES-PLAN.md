# Runbook — параллельная работа над слайсами в трёх сессиях

> **Назначение:** безопасно вести несколько слайсов одного эпика одновременно в разных
> сессиях Claude Code, не допуская влияния сессий друг на друга — ни на код (merge-конфликты),
> ни на контейнеры/БД (затирание данных). Механика изоляции: две оси (CLAUDE.md §16) плюс
> третья физическая машина.
>
> Родственные рунбуки: `STAGE-ENV.md` (ось B — stage-env), `GIT-WORKFLOW-COMMANDS.md`
> (ось A — worktree, ветки per stage), `DEPLOY.md`.

---

## 0. Модель изоляции (читать до запуска второй сессии)

Между двумя сессиями существует ровно два канала влияния. Оба должны быть закрыты:

1. **Рантайм/данные (контейнеры, БД, тома).** Закрывается физической изоляцией окружения:
   - На **VPS** — изолированный compose-проект через `scripts/stage-env.sh`
     (`STAGE_ENV=<name>` → `deploy/.env.<name>`, свой `STAGE_PROJECT`, свои тома/сеть/порты).
     Скрипт жёстко отвергает `STAGE_PROJECT=acme-platform|acme-platform-demo`.
   - На **dev-машине** — это отдельная физическая машина со своим Docker и своей БД;
     изоляция от VPS автоматическая (разные хосты).
2. **Код (общий git-remote).** Закрывается двумя средствами:
   - **git worktree** (ось A) — у каждой сессии своя рабочая директория и ветка.
   - **непересечение правимых файлов `src/`** — единственный реальный источник merge-конфликтов
     между слайсами. См. §3 (файловая матрица) и §5 (золотое правило).

Изоляция рантайма НЕ спасает от конфликта кода, и наоборот. Нужны обе.

## 2. Почему именно эта тройка (обоснование выбора — пример рассуждения)

- Ни один из трёх слайсов **не блокируется рейт-лимитом внешнего API**: Слайс 1 — чистая выгрузка
  из уже собранных данных; Слайс 2 — валидатор без сетевых вызовов; Слайс 3 бэкфиллит из данных,
  уже лежащих в БД, а не из внешнего API.
- Внешний блокер Слайса 3 снят — предыдущий PR смёржен в `main`.
- Три множества правимых файлов **не пересекаются** (§3) → чистые мержи.
- Слайсы, зависящие от живого внешнего API с известным rate-limit-риском, **сознательно не
  запускаются параллельно** — их нельзя довести до зелёного smoke/release (DoD §7), только
  «код в стол», пока лимит не снят.
- Слайсы, которые правят один и тот же файл или логически зависят друг от друга по мержу,
  **не запускаются одновременно** — секвенируются (§4).

## 3. Файловая матрица (гарантия неконфликтности)

```
Сессия A (Слайс 1):  src/module-a/{contracts,queries,server}.py
Сессия B (Слайс 2):  src/module-b/validator_service.py
Сессия C (Слайс 3):  src/module-c/{__main__,thresholds}.py, src/sync/cron_runner.py
```

Ни один файл не встречается дважды. Общий шаблон матрицы пересечений эпика (кто с кем
конфликтует при параллельном запуске — общий файл):

| Слайс | Правит | Конфликтует с | Блокер внешнего API |
|---|---|---|---|
| **1** экспорт | `src/module-a/{contracts,queries,server}.py` | — | нет |
| **2** валидатор | `src/module-b/validator_service.py` | — | нет |
| **3** бэкфилл | `src/module-c/{__main__,thresholds}.py`, `src/sync/cron_runner.py` | 4 | нет |
| **4** живой sync | `src/sync/{external_api,cron_runner}.py` | 3 | **да** |

## 4. Секвенирование (волны и зависимости)

```
Спайк(OQ-1) ─┐
Слайс 1 ──────┼──► решение о схеме экспорта → влияет на 2
Слайс 2   — независим, тупиковый (после мержа ничего не разблокирует)
Слайс 3   — независим, тупиковый; внешний блокер снят
Слайс 4   — код независим от 1/2/3; живой sync ждёт снятия rate-limit-блокера
```

- **Волна 1 (пример):** A=1, B=2, C=3 — параллельно, независимы.
- **Волна 2 (после мержа 1):** на освободившуюся дорожку — следующий зависимый слайс.
- **Волна 3 (строго последовательно):** слайсы, дерущиеся за общий файл (`cron_runner.py`
  в примере выше) — **между собой не параллелить**.

## 5. Золотое правило параллельного запуска

Перед стартом любой новой сессии проверь: **пересекаются ли множества правимых файлов `src/`
её слайса и всех уже идущих слайсов?** Если да — не параллель, секвенируй (§4).

Практическая проверка (в каждой worktree, после наметки правимых файлов):

```bash
git diff --name-only origin/main | sort > /tmp/lane-<X>.files
comm -12 /tmp/lane-A.files /tmp/lane-B.files   # пусто = дорожки не конфликтуют
```

## 6. Красные линии (нарушение = потеря данных / поломка чужой сессии)

**Общие для всех сессий:**
- `git fetch origin` до любого суждения о merge/branch/PR-состоянии (второй аккаунт мержит PR,
  локальные ref устаревают). Ветки создавай от `origin/main`, не от локального `main`.
- Одна стадия = один PR = человеческое одобрение. Не смешивать роли субагентов (CLAUDE.md §12).
- Не удалять файлы/ветки/тома, которые не создавал, без явного указания.
- Тесты гонять только против **своей** test-БД (integration-тесты стирают `public` schema).

**Дорожки A и B (VPS):**
- Все docker-действия — ТОЛЬКО `STAGE_ENV=<свой> bash scripts/stage-env.sh …`.
- НИКОГДА не выполнять docker-действий против проектов `acme-platform` / `acme-platform-demo` /
  чужого stage (`up/down/restart/build/down -v`). Это dev, demo и вторая дорожка — их данные боевые.
- stage-BI-инструмент НИКОГДА не поднимать с `--build` (пик RAM убьёт 8 GB коробку; откат ассетов).
- Экономь RAM: поднимай минимум сервисов (`up postgres` / `up postgres mcp-server`), не весь стек.

**Дорожка C (dev-машина):**
- Изолирована машиной — но НЕ подключаться к VPS-контейнерам/БД/сети и не выполнять
  деструктивных действий против удалённого стека.
- Локальный стек `acme-platform` — это твой dev; работать с ним штатно (`docker compose` локально)
  можно, но test-БД держать **отдельно** от dev-БД (не затирать dev-данные).
- Влияние на другие сессии — только через remote: строго ветка от `origin/main` + PR per stage,
  мерж по человеческому одобрению.

## Приложение A — шаблон промпта сессии

Копируй целиком, заполни шапку (СЛАЙС + ДОРОЖКА), запусти в сессии соответствующей машины.

````markdown
# ЗАДАЧА: провести слайс эпика «<название эпика>» через следующую стадию

## Параметры этой сессии (заполни перед запуском)
- СЛАЙС: <например «Слайс 1» / «Слайс 2» / «Слайс 3»>
- ДОРОЖКА: <A / B / C>   ← определяет ВСЕ твои изолированные ресурсы (таблица ниже)

Эпик-бриф: docs/features/<slug-эпика>/brief.md (PR #<N>).
Секвенирование и матрица конфликтов: docs/runbooks/PARALLEL-SLICES-PLAN.md.
Работай СТРОГО по CLAUDE.md (pipeline planner→architect→tdd→implementer→reviewer→release,
test-first, вертикальные слайсы, DoD §7). Не смешивай роли субагентов (§12).

## Твоя изолированная дорожка — используешь ТОЛЬКО эти ресурсы

| Ресурс             | Дорожка A (VPS)                  | Дорожка B (VPS)                   | Дорожка C (dev-машина)               |
|--------------------|----------------------------------|-----------------------------------|--------------------------------------|
| git worktree       | ~/projects/acme-platform-acc2    | ~/projects/acme-platform-laneB    | локальный dev-клон                   |
| Рантайм-механизм   | stage-env (stage-env.sh)         | stage-env (stage-env.sh)          | обычный локальный `docker compose`   |
| STAGE_ENV          | stage                            | stage2                            | — (не используется)                  |
| env-файл           | deploy/.env.stage                | deploy/.env.stage2                | deploy/.env (локальный dev)          |
| STAGE_PROJECT      | acme-platform-stage              | acme-platform-stage2              | acme-platform (локальный, на своей машине) |
| порты PG/BI/MCP    | 5436 / 8090 / 8767               | 5437 / 8091 / 8768                | 5434 / 8088 / 8765 (локальные)       |
| TEST_DATABASE_URL  | ...@127.0.0.1:5436/acme_platform_test | ...@127.0.0.1:5437/acme_platform_test | ...@127.0.0.1:5434/acme_platform_test |

Первым делом: подтверди дорожку, `pwd`, `git branch --show-current`. Если стоишь не на своей
дорожке — перейди в неё (§«Стартовый порядок»). Не начинай работу до этого.

## Красные линии (НАРУШЕНИЕ = потеря данных / поломка чужой сессии)
Общие:
- Перед суждением о merge/branch/PR — `git fetch origin`. Ветки — от `origin/main`.
- Тесты — ТОЛЬКО против СВОЕЙ test-БД (таблица). Integration-тесты стирают `public` schema.
- Одна стадия = один PR. Не мержь сам. Не удаляй чужое.

Дорожки A/B (VPS):
- ВСЕ docker-действия — ТОЛЬКО `STAGE_ENV=<твой> bash scripts/stage-env.sh ...`.
- НИКОГДА не трогай проекты `acme-platform`, `acme-platform-demo` и ЧУЖУЮ дорожку (up/down/restart/build).
- stage-BI-инструмент НИКОГДА не поднимай с `--build` (пик RAM). Поднимай минимум сервисов.

Дорожка C (dev-машина):
- НЕ подключайся к VPS-контейнерам/БД/сети; никаких деструктивных действий против удалённого стека.
- Локальный стек `acme-platform` — твой dev, работай штатно; test-БД держи ОТДЕЛЬНО от dev-данных.
- Влияние на другие сессии — только через remote: ветка от `origin/main` + PR per stage.

## Стартовый порядок
1. `git fetch origin`.
2. Встань в worktree своей дорожки. Если его нет (дорожка B, первый запуск на VPS):
   `git worktree add ~/projects/acme-platform-laneB origin/main`
   (дорожка C — обычный локальный клон, worktree создавать не требуется).
3. Создай ветку слайса от origin/main (GIT-WORKFLOW-COMMANDS.md):
   `git checkout -b feature/<slug-слайса>-stage<N> origin/main`
4. Определи текущую стадию слайса.

## Определи стадию (какой субагент запускать)
Смотри артефакты в docs/features/<slug-слайса>/ и состояние PR (`gh pr list`, `gh pr view`):
- нет brief.md / бриф не одобрен → planner (Stage 1)
- бриф одобрен, нет plan.md → architect (Stage 2)
- план одобрен, нет failing-тестов/test-matrix.md → tdd-engineer (Stage 3)
- тесты есть и падают → implementer (Stage 4)
- код зелёный, смёржен → reviewer (Stage 5)
- ревью одобрено → release-engineer (Stage 6)
Выполни РОВНО одну следующую стадию. Не перепрыгивай. Артефакт — по таблице CLAUDE.md §4.

## Когда поднимать окружение (экономия RAM на VPS — коробка 8 GB)
Контейнеры нужны НЕ на всех стадиях:
- Stage 1–3 (planner/architect/tdd): контейнеры НЕ нужны — только docs/тесты. Ничего не поднимай.
- Stage 4 (implementer): нужен postgres для quality-check.
  - A/B: `STAGE_ENV=<твой> bash scripts/stage-env.sh up postgres` (subset; для MCP-слайса
    добавь `mcp-server`). Создай в нём test-БД `acme_platform_test`, гоняй
    `TEST_DATABASE_URL=<свой> bash scripts/quality-check.sh`. По окончании — `... down`.
  - C: обычный локальный `docker compose up -d postgres`; отдельная локальная `acme_platform_test`.
- Stage 6 (release): полный стек + `SMOKE_MODE=staging bash scripts/smoke-check.sh` против СВОЕГО
  стека (A/B — никогда против acme-platform; C — против локального).
Перед первым up дорожки B — создать deploy/.env.stage2 из deploy/.env.stage.example
(STAGE_PROJECT=acme-platform-stage2, порты 5437/8091/8768, свои секреты), затем
`STAGE_ENV=stage2 bash scripts/stage-env.sh selfcheck` до `up`.

## Definition of done (CLAUDE.md §7) и правила PR
- Тесты зелёные против СВОЕЙ test-БД; итоговая строка pytest содержит `passed`, а не только
  `skipped` (иначе красный). Недоступность postgres — зафиксировать явно в PR.
- Follow-ups: в начале брифа гонять `scripts/followups.sh`; посторонние находки — `gh issue create`
  с меткой `followup,slice:<slug>`, не чинить на месте.
- PR: `--base` = ветка предыдущей стадии слайса (или main для Stage 1), `--head` — твоя ветка.
  НЕ мержить самому. Обновить docs/reference/DATA-MODEL.md, если тронут API-маппинг/метрики/ETL (§7).

## В конце
Отчитайся: какая стадия выполнена, какой PR открыт (номер/URL), какие контейнеры поднимал и
погасил ли, статус тестов (passed/skipped). Никаких «done» без прогона quality-check.
````
